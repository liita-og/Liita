## Part 2: Data Models (Single Source of Truth)

All models are defined in Dart. Native platform code communicates via JSON serialization of these structures.

### 2.1 `UserProfile`

```dart
class UserProfile {
  final String deviceId;     // UUID v4, generated once, stored in secure storage
  final String name;         // Max 20 characters
  final int age;             // 18–99
  final String seatNumber;   // e.g. "14C", max 4 chars
  final String occupation;   // Max 30 characters
  final String? photoHash;   // SHA-256 hash of profile photo bytes (nullable)
  final int version;         // Monotonic counter, increment on any profile change
  final String publicKey;    // Base64-encoded ECDH public key (P-256 curve)
  
  // Derived for BLE Scan Response packing (must fit in 31 bytes)
  // Format: name (20B) | age (1B) | seat (4B) | version (1B) = 26 bytes max
}
```

**Constraints:**
- `deviceId` is the canonical identifier across all packets and database records.
- `version` starts at 1 and increments by 1 on each profile update. Never reset.
- `publicKey` is advertised in-band. The private key never leaves the device.

### 2.2 `MeshPacket`

```dart
class MeshPacket {
  final String packetId;       // UUID v4 — used for deduplication
  final String originId;       // deviceId of the originating user
  final String destinationId;  // deviceId of the target user, or '*' for broadcast
  final int ttl;               // Time-to-live; starts at 8, decremented on each relay
  final String payloadType;    // See PayloadType enum below
  final String data;           // Base64-encoded payload (may be encrypted)
  final int timestamp;         // Unix timestamp in milliseconds (UTC)
}

enum PayloadType {
  wave,          // 'w' — wave intent signal
  waveAccept,    // 'a' — accepting a wave (creates match)
  text,          // 't' — encrypted chat message (AES-GCM)
  profileSync,   // 'p' — UserProfile JSON (unencrypted, signed)
  photoChunk,    // 'c' — photo chunk (see PhotoChunkPayload)
  broadcast,     // 'b' — flight public room message (unencrypted)
  ack,           // 'k' — delivery acknowledgement
}
```

**Relay Rules (implement exactly as specified):**
1. If `packet.packetId` is in the local dedup cache → **drop silently. Do not relay.**
2. If `packet.originId == localDeviceId` → **drop silently. Never relay your own originating packets.**
3. If `packet.destinationId == localDeviceId` → **consume. Do not relay.**
4. If `packet.destinationId == '*'` (broadcast) → **consume AND relay** (if `ttl > 1`).
5. If `packet.ttl <= 1` → **consume (if destination matches). Do not relay.**
6. Otherwise: **decrement ttl, add to dedup cache, re-broadcast via peripheral.**

### 2.3 `MatchEvent` (Append-Only Log)

```dart
class MatchEvent {
  final String eventId;      // UUID v4
  final String eventType;    // 'WAVE_SENT' | 'WAVE_RECEIVED' | 'MATCH_CREATED' | 'BLOCKED'
  final String actorId;      // deviceId of who performed the action
  final String targetId;     // deviceId of who the action targets
  final int timestamp;       // Unix timestamp in milliseconds (UTC)
}
```

**Match Derivation Logic:** A match exists between `deviceA` and `deviceB` if and only if:
- A `WAVE_SENT` event exists with `actorId = deviceA, targetId = deviceB`
- AND a `WAVE_SENT` event exists with `actorId = deviceB, targetId = deviceA`
- AND no `BLOCKED` event exists for either direction

### 2.4 `ChatMessage`

```dart
class ChatMessage {
  final String messageId;    // UUID v4
  final String matchId;      // Composite key: sorted(deviceA, deviceB) joined by ':'
  final String fromId;       // deviceId of sender
  final String toId;         // deviceId of recipient
  final String ciphertext;   // Base64-encoded AES-GCM encrypted content
  final String nonce;        // Base64-encoded 12-byte AES-GCM nonce
  final int timestamp;       // Unix timestamp in milliseconds (UTC)
  final bool delivered;      // Set to true on receipt of 'k' ACK packet
  final bool isRead;         // Set to true when chat screen is opened
}
```

### 2.5 `PhotoChunkPayload`

```dart
class PhotoChunkPayload {
  final String photoHash;    // SHA-256 of the complete photo (session identifier)
  final int chunkIndex;      // 0-based chunk index
  final int totalChunks;     // Total number of chunks in this photo
  final String data;         // Base64-encoded chunk bytes (max 400 bytes per chunk)
}
```

**Photo Transmission Rules:**
- Photos are chunked into 400-byte segments before Base64 encoding.
- Receiver buffers chunks by `(photoHash, chunkIndex)`.
- When `received_chunks.length == totalChunks`, reassemble, verify SHA-256 hash, write to disk.
- Incomplete transmissions are abandoned after 5 minutes with no new chunks received.

## Part 3: Database Schema (SQLite via `sqflite`)

### Schema Version: 1

```sql
-- User profile cache (profiles of all discovered peers)
CREATE TABLE profiles (
  device_id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  age INTEGER NOT NULL,
  seat_number TEXT NOT NULL,
  occupation TEXT NOT NULL,
  photo_hash TEXT,
  version INTEGER NOT NULL DEFAULT 1,
  public_key TEXT NOT NULL,
  last_seen INTEGER NOT NULL  -- unix timestamp ms
);

-- Append-only match event log
CREATE TABLE match_events (
  event_id TEXT PRIMARY KEY,
  event_type TEXT NOT NULL,
  actor_id TEXT NOT NULL,
  target_id TEXT NOT NULL,
  timestamp INTEGER NOT NULL
);
CREATE INDEX idx_match_events_actors ON match_events(actor_id, target_id);

-- Encrypted chat messages
CREATE TABLE messages (
  message_id TEXT PRIMARY KEY,
  match_id TEXT NOT NULL,
  from_id TEXT NOT NULL,
  to_id TEXT NOT NULL,
  ciphertext TEXT NOT NULL,
  nonce TEXT NOT NULL,
  timestamp INTEGER NOT NULL,
  delivered INTEGER NOT NULL DEFAULT 0,  -- boolean
  is_read INTEGER NOT NULL DEFAULT 0     -- boolean
);
CREATE INDEX idx_messages_match ON messages(match_id, timestamp);

-- Photo chunk buffer
CREATE TABLE photo_chunks (
  photo_hash TEXT NOT NULL,
  chunk_index INTEGER NOT NULL,
  total_chunks INTEGER NOT NULL,
  data TEXT NOT NULL,        -- base64 chunk
  received_at INTEGER NOT NULL,
  PRIMARY KEY (photo_hash, chunk_index)
);

-- Packet deduplication cache (in-memory preferred, SQLite as backup)
CREATE TABLE packet_dedup (
  packet_id TEXT PRIMARY KEY,
  received_at INTEGER NOT NULL  -- unix timestamp ms; prune entries older than 10 minutes
);
```

**Migration Policy:** All schema changes must be handled via `sqflite`'s `onUpgrade` callback. Never drop tables in migrations. Add columns with `ALTER TABLE ... ADD COLUMN` and provide default values.

### 4.2 `MeshService` Abstract Interface

```dart
abstract class MeshService {
  // Control
  Future<void> startMesh(UserProfile localProfile);
  Future<void> stopMesh();
  
  // Streams — all consumed by AppController
  Stream<UserProfile> get discoveredPeers;   // emits on each new/updated profile seen
  Stream<MeshPacket> get incomingPackets;    // emits pre-relay packets for local consumption
  
  // Outbound
  Future<void> sendPacket(MeshPacket packet);
  
  // State
  bool get isRunning;
  Stream<int> get activePeerCount;
}
```

**Critical Constraint:** `MeshService` implementations must never call `sendPacket` internally for relay — relay logic happens inside the native layer. Dart-side `sendPacket` is only for *originating* new packets.

### 4.3 `CryptoService` Interface

```dart
abstract class CryptoService {
  // Key Management
  Future<AsymmetricKeyPair> generateKeyPair();       // ECDH P-256
  Future<String> exportPublicKey(PublicKey key);     // Base64 DER
  Future<PublicKey> importPublicKey(String base64);
  
  // ECDH
  Future<Uint8List> deriveSharedSecret(
    PrivateKey myPrivateKey,
    PublicKey theirPublicKey,
  );
  
  // AES-GCM (key is 32-byte derived secret)
  Future<EncryptedPayload> encrypt(String plaintext, Uint8List key);
  Future<String> decrypt(EncryptedPayload payload, Uint8List key);
}

class EncryptedPayload {
  final String ciphertext; // Base64
  final String nonce;      // Base64, 12 bytes
}
```

**Implementation Notes:**
- Use `pointycastle` package for all crypto primitives in Dart.
- Shared secrets are derived using ECDH and then hashed with HKDF-SHA256 to produce the final 32-byte AES key.
- Store derived shared secrets in the platform keychain (iOS `Keychain`, Android `EncryptedSharedPreferences`) keyed by `matchId`.
- The private key is generated once at install and stored in the platform keychain. It is never exported or transmitted.

### 5.3 Packet Broadcast Over BLE

Because BLE advertisement payloads are limited to 31 bytes, `MeshPacket` objects are not transmitted in advertisement packets. The relay protocol uses a **GATT Write** to the `MESH_PROFILE_CHAR` of the central's GATT server.

**Relay Flow:**
```
Device A (originator) → Device B (relay) → Device C (destination)

Step 1: A encodes MeshPacket as JSON → compresses with gzip → Base64
Step 2: A writes compressed packet to Device B's GATT characteristic (via Write Without Response)
Step 3: B receives write → decompresses → validates packetId dedup → decrements ttl
Step 4: B writes to all connected peripherals with ttl > 1
Step 5: C receives, destinationId matches → consume, emit to Flutter layer, send ACK
```

**Size Budget for MeshPacket (compressed):**
- Text message: ~180 bytes compressed. Fits in a single GATT write (512 byte MTU on modern devices).
- Wave/Accept: ~80 bytes. Trivially small.
- Profile sync: ~300 bytes compressed.
- Photo chunks: Each chunk is 400 bytes. Send as multiple writes with `chunkIndex` tracking.

### 6.3 Incoming Packet Handler

This is the central routing logic, implemented in `AppController`:

```dart
void _handleIncomingPacket(MeshPacket packet) {
  switch (packet.payloadType) {
    case PayloadType.wave:
      _handleWave(packet);
    case PayloadType.waveAccept:
      _handleWaveAccept(packet);
    case PayloadType.text:
      _handleText(packet);
    case PayloadType.profileSync:
      _handleProfileSync(packet);
    case PayloadType.photoChunk:
      _handlePhotoChunk(packet);
    case PayloadType.broadcast:
      _handleBroadcast(packet);
    case PayloadType.ack:
      _handleAck(packet);
  }
}

void _handleWave(MeshPacket packet) async {
  await _db.insertMatchEvent(MatchEvent(
    eventType: 'WAVE_RECEIVED',
    actorId: packet.originId,
    targetId: localDeviceId,
    timestamp: DateTime.now().millisecondsSinceEpoch,
  ));
  await _notifications.showWaveNotification(senderName: _getPeerName(packet.originId));
  
  // Check if we previously waved at this person → auto-create match
  final weWavedFirst = await _db.hasWaveSent(localDeviceId, packet.originId);
  if (weWavedFirst) {
    await _createMatch(packet.originId);
  }
}

Future<void> _createMatch(String peerId) async {
  await _db.insertMatchEvent(MatchEvent(eventType: 'MATCH_CREATED', ...));
  // Exchange public keys if not already stored
  await _initiateKeyExchange(peerId);
}
```

## Part 7: Security Specification

### 7.1 Threat Model

| Threat | Mitigation |
|---|---|
| Passive eavesdropping of BLE broadcasts | AES-GCM encryption on all 't' packets |
| Replay attacks | Unique `packetId` (UUID v4) + dedup cache |
| Profile spoofing | Profiles are signed with sender's private key |
| Relay node reading message content | End-to-end encryption; relay nodes see only ciphertext |
| Packet flooding / DoS | TTL decrement; max 8 hops; dedup cache rejects duplicates |
| Key impersonation | Public keys are bound to `deviceId`; trust-on-first-use model |

### 7.2 Key Exchange Protocol

```
1. Alice sends Wave to Bob (packet contains Alice's publicKey in data field)
2. Bob receives Wave, extracts Alice's publicKey, stores in profiles table
3. Bob sends WaveAccept to Alice (packet contains Bob's publicKey in data field)
4. Alice receives WaveAccept, extracts Bob's publicKey
5. Both derive: sharedSecret = ECDH(myPrivateKey, theirPublicKey)
6. Both derive: aesKey = HKDF-SHA256(sharedSecret, salt="liita-v1", length=32)
7. aesKey is stored in platform keychain keyed by matchId
8. All subsequent 't' packets encrypted with this aesKey
```

### 7.3 Block List Implementation

When Device A blocks Device B:
1. Insert `BLOCKED` event into `match_events`.
2. Add `deviceB.deviceId` to local block list (stored in `EncryptedSharedPreferences` / iOS Keychain).
3. `MeshService` native layer checks block list before relaying any packet. Blocked device packets are **dropped at the hardware callback level**, not Dart level.

## Part 9: Regression Contract

> **Every AI agent session working on `MeshService`, `CryptoService`, or `DatabaseService` must acknowledge and preserve these contracts before writing any code.**

1. **Dedup First:** Any incoming BLE packet is checked against the dedup cache *before* any other processing. Period.
2. **Never Relay Own Origin:** A device never relays a packet where `originId == localDeviceId`.
3. **Encrypt Before Transmit:** A `'t'` (text) type MeshPacket's `data` field is *always* AES-GCM encrypted before being handed to `MeshService.sendPacket()`.
4. **Database on Background Isolate:** All `DatabaseService` operations use `compute()` or a dedicated Dart isolate. Zero database I/O on the main UI thread.
5. **Stream, Don't Poll:** UI layers read peer/match/message state via Streams/Providers. No `Timer`-based polling of the database.
6. **Private Key Never Leaves Keychain:** The ECDH private key is generated in and read from the platform keychain only. It is never serialized to JSON, logged, or placed in a MeshPacket.
7. **TTL Is Authoritative:** Relay decisions are made by the native layer. The Dart layer does not implement relay logic.
8. **Schema Migrations Are Additive:** Database schema changes only add tables or columns. No destructive migrations.

## Appendix A: BLE Payload Byte Budget

### Advertisement Packet (31 bytes max)
```
[Flags: 3 bytes] [ServiceUUID: 18 bytes] [DeviceId: 8 bytes] [Reserved: 2 bytes]
```

### Scan Response Packet (31 bytes max — Android only)
```
[Name: 20 bytes] [Age: 1 byte] [Seat: 4 bytes] [ProfileVersion: 1 byte] [Reserved: 5 bytes]
```

### GATT Write MTU (varies by device)
```
Minimum guaranteed:  23 bytes (subtract 3 for ATT header = 20 bytes usable)
Modern devices:     512 bytes (use GATT MTU negotiation: requestMtu(512))
Recommended:        Fragment any payload > 400 bytes
```
